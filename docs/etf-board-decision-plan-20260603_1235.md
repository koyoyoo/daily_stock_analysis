# ETF板块决策板计划文档

本文件唯一职责：定义 `ETF板块决策板` 的本地落地计划、默认边界、评分框架与分阶段实施路径，供后续产品设计与开发执行使用。

## 1. 背景与目标

当前首页的核心能力是单标的诊股，能够输出个股趋势、技术面、新闻摘要、部分板块关联信息，但还不能稳定回答下面这个交易问题：

- 当前某个板块对应的 ETF，应该 `买入`、`持有`、`减仓` 还是 `清仓`
- 为什么给出这个动作
- 这个动作背后的强弱，能否被量化成一个清晰评分

本计划文档的目标是定义一套面向 `A股行业/主题 ETF` 的板块级决策框架，把 `指数`、`题材龙头`、`板块情绪`、`核心个股`、`资金与事件` 聚合成一份可解释、可追踪、可扩展的 ETF 决策结果。

## 2. 默认边界

以下边界已经过确认，作为当前 MVP 的冻结约束。若后续业务确认不同，应先更新本文件，再进入开发。

- 市场范围：只覆盖 `A股行业/主题 ETF`
- 决策周期：只覆盖 `日线级波段决策`，不做分钟级和日内超短
- 诊断对象：一个板块默认绑定一个主 ETF
- 龙头样本：每个板块默认分析 `3` 只核心股票
- 输出目标：给出 `评分 + 动作 + 理由 + 风险`
- 动作集合：`买入 / 持有 / 减仓 / 清仓`
- 配置方式：第一版接受 `人工维护板块映射`
- 资金优先级：固定为 `ETF份额变化 -> 龙头成交额/换手/量比 -> 板块成交额变化`

## 3. 非目标

本计划不包含以下内容：

- 不在第一版自动识别所有板块龙头
- 不在第一版支持 `港股 / 美股 / 跨市场 ETF`
- 不在第一版支持一板块多 ETF 智能优选
- 不在第一版支持分钟级盘口、逐笔、超短情绪博弈
- 不在第一版替代现有个股诊股
- 不在第一版引入新的数据库迁移或复杂配置中心

## 4. 核心问题定义

系统需要回答的不是“ETF 会不会涨”，而是以下四个可执行问题：

1. 当前板块是否具备持续性，而不是单日脉冲。
2. 板块上涨是否由龙头、中军、跟风形成共振，而不是单一权重硬拉。
3. 当前 ETF 的上涨逻辑是趋势型、情绪型，还是事件型。
4. 在当前时点，最合理的交易动作是买入、持有、减仓还是清仓。

## 6. 评分框架

当前版本固定采用 `100 分制`，并通过固定维度权重保证可解释性。

### 6.1 维度与权重

- `指数强弱`：25 分
- `龙头共振`：30 分
- `板块情绪`：20 分
- `资金状态`：15 分
- `事件催化`：10 分

### 6.2 动作阈值

当前版本固定使用以下阈值：

- `80-100`：买入
- `65-79`：持有
- `50-64`：减仓
- `<50`：清仓

### 6.3 置信度规则

置信度不直接由总分决定，而由“数据覆盖完整度 + 结论一致性”共同决定：

- `高`：5 个维度中至少 4 个维度为 `available`，且不存在核心信号冲突
- `中高`：核心维度完整，存在轻微冲突，或存在单个 `partial`
- `中`：存在多个 `partial` / `estimated`，但主信号仍可解释
- `低`：核心字段缺失，或指数、龙头、情绪出现显著冲突

## 7. 维度定义

### 7.1 指数强弱

目标：判断该板块所对应的指数是否具备趋势基础。

建议观察项：

- 行业指数相对沪深 300 或中证全指的强弱
- 5 / 10 / 20 / 60 日均线结构
- 量价配合是否健康
- 是否刚突破、持续趋势，还是高位钝化
- 回撤是否仍处于强势结构

建议输出：

- 指数分数
- 趋势状态
- 是否处于追高风险区

### 7.2 龙头共振

目标：判断板块核心个股是否形成同向驱动。

默认规则：

- 每个板块维护 `1 个绝对龙头 + 2 个核心跟随股`
- 分析每只个股的趋势、量能、筹码、事件状态
- 关注“龙头强、中军稳、跟风补涨”是否成立

建议观察项：

- 龙头是否处于上升趋势
- 第二、第三核心股是否同步走强
- 是否出现“只拉龙头，不带板块”的虚强
- 龙头是否放量滞涨、冲高回落或利好兑现

### 7.3 板块情绪

目标：衡量板块热度、扩散度和短线持续性。

建议观察项：

- 板块涨跌家数占比
- 前排个股一致性
- 板块热度是否升温或退潮
- 是否存在高低切换、补涨扩散
- 同题材内部是否出现明显分化

说明：

- 第一版允许情绪维度采用 `规则指标 + LLM 归纳` 混合方式
- 若缺少稳定量化数据，必须降级并降低置信度，不能伪装成强结论

### 7.4 资金状态

目标：判断增量资金是否仍在板块内活跃。

固定优先级：

1. ETF 份额变化
2. 龙头股成交额、换手、量比变化
3. 板块成交额变化

建议观察项：

- ETF 份额是否持续增加
- 板块成交额是否明显放大
- 龙头是否出现健康放量
- 是否只有情绪冲高，没有资金延续

### 7.5 事件催化

目标：判断当前驱动是持续催化还是一次性脉冲。

建议观察项：

- 政策催化是否明确
- 行业景气逻辑是否强化
- 龙头股财报、公告、订单、产能等事件是否正向
- 事件是否已经被价格充分定价

## 9. 决策引擎设计

建议把板块能力拆为独立服务，避免污染现有单股诊股服务。

### 9.1 后端模块

建议目录：

```text
src/services/board_diagnosis/
  __init__.py
  board_registry.py
  models.py
  index_signal_analyzer.py
  leader_signal_analyzer.py
  sentiment_signal_analyzer.py
  capital_signal_analyzer.py
  event_signal_analyzer.py
  decision_scorer.py
  explanation_renderer.py
```

模块职责：

- `board_registry.py`
  - 维护板块、ETF、指数、龙头股映射
- `index_signal_analyzer.py`
  - 输出指数维度评分与理由
- `leader_signal_analyzer.py`
  - 输出龙头共振维度评分与理由
- `sentiment_signal_analyzer.py`
  - 输出板块情绪评分与理由
- `capital_signal_analyzer.py`
  - 输出资金维度评分与理由
- `event_signal_analyzer.py`
  - 输出事件催化维度评分与理由
- `decision_scorer.py`
  - 汇总维度分数，生成总分、动作、置信度
- `explanation_renderer.py`
  - 生成前端和报告所需的可解释文本

### 9.2 前端模块

建议目录：

```text
apps/dsa-web/src/features/board-diagnosis/
  api/
  components/
  hooks/
  types/
```

建议首屏卡片：

- `ETF结论卡`
- `指数状态卡`
- `龙头共振卡`
- `板块情绪卡`
- `资金与事件卡`
- `决策解释卡`

## 10. 与现有系统的关系

### 10.1 复用项

当前系统可复用以下能力：

- 个股基础分析链路
- 历史记录与报告展示框架
- 新闻拉取与摘要能力
- 技术面与筹码已有分析基础
- Web 报告卡片体系和任务流转机制

### 10.2 必须新增项

当前系统缺少以下关键能力，必须新增：

- 板块对象与映射配置
- 板块级评分模型
- 多标的聚合分析
- ETF 专属决策对象
- 板块情绪与资金的明确字段契约

## 11. 数据来源建议

第一版优先复用现有数据源，缺失时显式降级。

第一版数据来源分层：

- `指数层`
  - 现有行情接口或日线数据源
- `龙头层`
  - 现有个股分析链路
- `情绪层`
  - 板块涨跌家数、板块排行、新闻热度、LLM 归纳
- `资金层`
  - ETF 份额、ETF 成交额、龙头成交额、换手、量比
- `事件层`
  - 新闻搜索、公告搜索、财报窗口、政策催化

Fail Fast 原则：

- 如果核心指数数据缺失，直接降低总置信度
- 如果龙头样本缺失超过一半，拒绝给出高置信度买入结论
- 如果资金数据不可得，不允许伪造“主力持续流入”类文案

## 12. 分阶段实施计划

### Phase 0：计划与契约

目标：

- 固化板块对象
- 固化评分框架
- 固化前后端字段契约

交付：

- 本计划文档
- 板块配置样例
- 决策输出结构草案

验证：

- 文档审阅
- 字段设计评审

### Phase 1：后端 MVP

目标：

- 在后端输出一份可用的板块诊断结果

范围：

- 手工维护 3-5 个板块样例
- 每个板块分析 1 个指数 + 3 只核心股
- 输出总分、动作、理由、风险
- 暴露只读 API 能力，不在本阶段承诺 Web 入口

建议 API：

- `GET /api/v1/board-diagnosis/boards`
- `GET /api/v1/board-diagnosis/{board_key}`

建议验证命令：

```bash
py -m pytest tests -k board_diagnosis
py -m py_compile src/services/board_diagnosis/*.py
```

### Phase 2：前端展示 MVP

目标：

- 在 Web 端落地板块决策卡片

范围：

- 新增专题入口或在首页增加板块诊断入口
- 展示评分、动作、五维得分与理由

建议验证命令：

```bash
cd apps/dsa-web
pnpm lint
pnpm build
```

### Phase 3：数据增强

目标：

- 提升板块情绪、资金、事件维度的客观性

范围：

- 引入更多板块强弱字段
- 优化 ETF 份额与成交额数据
- 增加同板块多 ETF 对比能力

## 13. MVP 板块建议

建议第一批只做 5 个高辨识度板块：

- 新能源
- 白酒
- 机器人
- 半导体
- 证券

原因：

- 市场认知清晰
- 龙头股映射容易维护
- ETF 代表性较强
- 用户能快速验证结论是否符合交易直觉

## 14. 风险与防腐策略

### 14.1 主要风险

- 板块定义漂移，导致 ETF、指数、龙头映射失真
- 情绪与资金数据缺失时，LLM 过度解释
- 评分规则过于复杂，最终不可维护
- 第一版同时支持太多板块，验证成本过高

### 14.2 防腐策略

- 第一版固定显式配置，不做自动板块识别
- 每个维度必须输出“证据摘要”
- 低质量数据必须降低置信度
- 评分权重固定，不做用户侧自由配置
- 每次扩展板块时先补配置，再接入 UI

## 15. 已确认项

以下事项已经确认，并作为当前 MVP 的固定契约：

- 第一版只做 `A股 ETF`
- 每个板块固定分析 `3` 只核心股
- 动作阈值固定采用 `80 / 65 / 50`
- 资金维度固定优先使用 `ETF份额变化 + 龙头成交额/换手/量比 + 板块成交额变化`
- 第一版先落 `后端只读 API`，前端入口在 Phase 2 决定是 `首页扩展` 还是 `独立专题页`

## 16. 推荐实施顺序

建议严格按以下顺序推进：

1. 固化板块配置模型
2. 固化评分契约与输出对象
3. 先完成后端聚合诊断
4. 再接入前端卡片展示
5. 最后补充历史记录、回放和策略迭代

## 17. 本次结论

`ETF板块决策板` 不应继续作为现有单股诊股的临时补丁来实现，而应作为一条独立分析能力落地。第一版的正确方向不是追求覆盖所有板块，而是通过少量高辨识度板块验证“指数 + 龙头 + 情绪 + 个股 + 资金/事件 -> ETF动作决策”这套框架是否真实提升交易判断质量。
